## **1.1 모델링의 이해**

-----

### **1. 모델링**

모델링(modeling)이란 현실세계의 대상으로 일종의 모델을만드는 것을 의미한다.

```text
현실 세계
    │
    ▼
구성요소 분해
    │
    ▼
꼭 필요한 요소 추출
    │
    ▼
데이터 모델
```

우리가 모델링 하는 이유 중 하나가 업무의 흐름(업무의 대상)을 가시화하고 명세화하여 이를 설계, 개발 또는 시스템 관리에 사용하기 위한 것이기 때문이다.

모델링이 추상화를 기본으로 한다.

`추상화`
- 단순화
- 축약
- 구성요소

-----

### **2. 모델링의 특징**

#### **1. 추상화(Abstraction)**

  대상의 주요 특징을 추출하여 일정한 형식으로 표현한다.

#### **2. 단순화(Simplification)**

  복잡한 현실세계를 그대로 표현하지 않고 보다 단순하개 표현한다.

  UML(Unified Modeling Language)이나 ERD(Entity Relationship Diagram)와 같이 제한된 언어나 **약속된 표기법**을 사용하여 누구나 이해하기 쉽도록 한다.
  
#### **3. 명확화(Clarity)**

  모델링의 결과는 보는 사람에 따라 서로 다르게 해석되지 않고 대상을 명확하게 이해할수 있도록 모호함이 없게 기술한다.

----

### **데이터 모델링 시 유의사항**

#### **중복 최소화**

  여러 곳에 같은 정보를 중복하여 저장하지 않아야 한다.
  
#### **비유연성 최소화**

  데이터의 정의와 데이터의 사용 프로세서를 분리

#### **비일관성 최소화**

  데이터 간의 상호 연관관계를 명확하게 정의하여 데이터가 일관성 있게 유지

-----

### **3. 모델링의 세가지 관점**

#### **1. 데이터 관점(대상)**

  업무를 구성하는 데이터에 집중하여, 어떤 데이터들이 서로 관계를 맺고 사용되는지를 모델링하는 것이다.

#### **2. 프로세스 관점(처리방법)**

  업무가 실제로 처리하는 일이 어떻게 처리괴는지를 모델링하는 것이다.

#### **3. 데이터와 프로세스 상관 관점(대상과 처리방법의 상관관계)**

  업무를 구성하는 데이터와 프로세스가 서로 어떻게 관계를 맺고 영향을 주고받는지를 모델링 하는 것이다. CRUD(Create, Read, Update, Delete) 분석 기반으로 한다.

------

### **4. 모델링의 세 가지 단계**

#### **1. 개념적 데이터 모델링**

  업무와 개념 중심으로 포괄적인 수준에서 모델링을 수행한다.

  이 단계에서 엔티티, 속성 도출한다.ㅣ
  
#### **2. 논리적 데이터 모델링**

  키, 속석, 관계 등을 표현하며 재사용성이 높다.

  정규화를 통해 중복 데이터를 최소화한다.

  이 단계에서 식별자를 도출하고 관계를 정의한다.
  
#### **3. 물리적 데이터 모델링**
(가장 구체적인 표현)

  DBMS의 성능이나 보안, 가용성 등을 고려하여 설계하는 것이다.

  테이블, 인덱스, 함수 등을 생성한다.
  
추상화 레벨 (CDM > LDM > PDM)
```text
개념적 데이터 모델링
    │
    ▼
논리적 데이터 모델링
    │
    ▼
물리적 데이터 모델링
```

------

### **ANSI-SPARC에서 정의한 3단계 스키마 구조**

데이터 독립성을 보장하기 위한 설계방법을 제시

#### **1. 외부 스키마**

  사용자 관점, 사용자 또는 애플리케이션이 바라보는 데이터베이스 스키마를 정의한다.

  다중 사용자 뷰를 제공한다.

#### **2. 개념 스키마**

  설계자 관점, 모든 사용자가 바라보는 데이터베이스 스키마를 통합하여 나타낸다.

  통합된 뷰를 제공한다.

#### **3. 내부 스키마**

  개발ㅈ 관점, 디스크나 메모리 상의 물리적, 실질적 저장구조를 나타내며, 테이블, 칼럼, 인덱스 등을 정의한다.

  물리적 뷰를 제공한다.

-----

### **6. ERD**
  데이터베이스를 구성하는 데이터의 논리적 구성요소를 엔터티로 정의하고,

  엔터티를 간의 관계를 특별한 표기법으로 나타냄으로써 스키마를 설계하난 방법을 제공한다.

  - 엔터티: 엔터티는 박스로 나타낸다. 엔터티 이름은 박스 상단에 표시한다.
  - 카디널리지: 까치발이라고 부르는 관계의 카디널리티는 O는 0, 세로 실선은 1, 세 개의 까치발 모양은 다수(2 이상)을 의미한다.
  - 속성: 속성은 박스 안에 리스트 형태로 표현한다. 주식별자(PK)와 일반 속성은 칸막이로 구분된다.
  - 관계: 관계는 박스로 연결한 선으로 표현한다. 식별자 관계는 실선, 비식별자 관계는 점선으로 표현한다.

#### **ERD 작성순서**

1. 엔터티를 도출한다.
2. 도출한 엔터티를 적절하게 베치한다. 중요한 엔터티는 왼쪽 상단에 베치한다.
3. 엔터티 간의 관계를 표현한다
4. 관계명을 기술한다. (행위 관계, 존재 관계를 표현)
5. 관계의 참여도를 기술한다 (일대일, 이대다, 다대다 표현)
6. 관계의 필수/선택 여부를 가슐한다 (Null 값을 가질수 있는지 여부)
