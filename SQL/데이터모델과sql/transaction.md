## **모델이 표현하는 트랜잭션의 이해**

데이터베이스에서 데이터를 삽입, 수정, 삭제할 때는 작업이 **원자적(Atomic)** 으로 하나의 단위로 실행되어야 전체 데이터 정합성이 유지된다.  
여기서 원자적이란 **더 이상 쪼갤 수 없는 단위**를 의미한다.

즉, 작업은 **완전히 수행되거나(성공) 전혀 수행되지 않아야(실패)** 하며, 중간에 미완료 상태로 끝나면 안 된다.  
데이터 정합성이 중요하므로 데이터 읽기/쓰기 작업은 트랜잭션 단위로 처리되어야 한다.

---

### **트랜잭션 관련 SQL 명령어 (TCL)**

| 명령어 | 설명 |
| --- | --- |
| COMMIT | 작업을 정상 완료하고 데이터베이스에 최종 반영 |
| ROLLBACK | 작업을 취소하고 이전 상태로 복구 |
| SAVEPOINT | 트랜잭션 내 부분 롤백을 위한 저장점 지정 |

---

## **1. 트랜잭션의 개념**

트랜잭션(Transaction)은 **한 번에 수행되어야 하는 논리적 작업 단위**이다.  
데이터베이스는 트랜잭션 단위로 데이터 조작이 발생하며, 하나의 일관된 상태에서 다른 일관된 상태로 전이한다.

---

### **2. 트랜잭션의 특성(ACID)**

| 특성 | 설명 |
| --- | --- |
| 원자성 (Atomicity) | 하나의 트랜잭션 연산은 **All or Nothing**으로 모두 수행되거나 전혀 수행되지 않아야 함 |
| 일관성 (Consistency) | 트랜잭션 전후에 데이터베이스 규칙/정합성이 유지되어야 함 |
| 고립성 (Isolation) | 동시에 실행되는 트랜잭션 간 간섭 없이 독립적으로 수행되어야 함 |
| 영속성 (Durability) | 커밋된 결과는 장애 발생 후에도 영구적으로 보존되어야 함 |

---

### **3. 격리수준(Isolation Level)이 낮을 때의 문제점**

| 현상 | 설명 |
| --- | --- |
| Dirty Read | 커밋되지 않은 데이터를 다른 트랜잭션이 읽어 발생하는 불일치 |
| Non-Repeatable Read | 같은 트랜잭션에서 같은 조회를 두 번 했을 때 결과가 달라지는 현상(수정/삭제 영향) |
| Phantom Read | 같은 트랜잭션에서 같은 조건 조회 시, 중간 삽입으로 이전에 없던 행이 나타나는 현상 |

---

### **4. 이상 현상 요약 비교**

| 구분 | 발생 원인 | 결과 |
| --- | --- | --- |
| Dirty Read | 미커밋 데이터 읽기 | 잘못된 데이터 기반 처리 |
| Non-Repeatable Read | 조회 사이 데이터 변경/삭제 | 동일 조회 결과 불일치 |
| Phantom Read | 조회 사이 조건에 맞는 신규 행 삽입 | 결과 집합 행 수 변화 |

```text
트랜잭션 A (조회)
   ├─ 1차 조회
   ├─ (트랜잭션 B가 중간에 변경/삽입)
   └─ 2차 조회
       ├─ 값이 달라짐 → Non-Repeatable Read
       └─ 행이 늘어남 → Phantom Read
```
