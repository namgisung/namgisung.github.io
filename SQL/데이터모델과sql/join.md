## **관계와 조인의 이해**

정규화를 통해 분해된 테이블들은 서로 관계(Relationship)를 맺게 된다.  
즉, 원래 테이블의 식별자 중 일부를 분해된 테이블들이 공통으로 가지며, 하나가 다른 하나를 참조하는 구조가 된다.

- **참조하는 쪽**: 자식 테이블(Child)
- **참조되는 쪽**: 부모 테이블(Parent)

정규화로 분해된 테이블은 조회 시 다시 결합해야 할 때가 많고, 이때 공통 식별자를 기준으로 두 테이블을 결합하는 연산이 **조인(Join)** 이다.

조회 시 조인이 많아지면 성능이 저하될 수 있으므로, 필요 시 정규화와 반대로 테이블을 다시 합치거나 중복을 허용해 조회 성능을 높일 수 있는데 이를 **반정규화(Denormalization)** 라고 한다.

---

### **1. 조인을 하는 이유**

| 항목 | 내용 |
| --- | --- |
| 조인 목적 | 서로 다른 테이블에 나뉜 속성을 한 번에 조회 |
| 조인 전제 | 테이블 간 관계(매핑 가능한 키)가 존재해야 함 |
| 대표 매핑키 | PK/FK, 또는 업무적으로 일치하는 공통 식별자 |

---

### **2. 반정규화 개요**

| 항목 | 내용 |
| --- | --- |
| 정의 | 데이터 중복 허용 또는 그룹핑으로 조회 성능을 높이는 기법 |
| 기대 효과 | 조회 성능 향상, 개발/운영 단순화 |
| 주의점 | 데이터 정합성(일관성) 저하 가능성 |

#### 반정규화 전 검토 대안

| 대안 | 설명 |
| --- | --- |
| 뷰(View) 활용 | 쿼리 단순화 및 재사용성 향상 |
| 테이블 조정 | 테이블 구조/분할/병합 재검토 |
| 인덱스 조정 | 액세스 경로 최적화 |
| 클러스터링 적용 | 물리적 저장 최적화 |
| 애플리케이션 처리 | DB 외부 로직으로 일부 연산 분산 |

> 반정규화는 정합성을 낮출 수 있으므로, 위 대안을 충분히 검토한 뒤 적용한다.

---

### **3. 성능 데이터 모델링**

분석/설계 단계부터 데이터베이스 성능을 고려해 모델링을 수행하는 개념이다.

| 순서 | 절차 |
| --- | --- |
| 1 | 데이터 모델링 시 정규화를 정확히 수행 |
| 2 | 데이터베이스 용량 산정 수행 |
| 3 | 데이터베이스 트랜잭션 유형 파악 |
| 4 | 용량/트랜잭션 유형에 따라 반정규화 수행 |
| 5 | 이력 모델, PK/FK, 슈퍼타입/서브타입 조정 |
| 6 | 성능 관점에서 데이터 모델 검증 |

---

## **4. 관계(Relationship)**

관계는 엔터티(또는 테이블) 간의 논리적 연관성을 의미한다.

### 4.1 관계의 분류

| 분류 기준 | 유형 | 설명 |
| --- | --- | --- |
| 의미적 분류 | 존재적 관계 | 존재 자체로 연관 |
| 의미적 분류 | 행위적 관계 | 이벤트/행위로 연관 |
| 형식적 분류 | 식별자 관계 | 부모 식별자가 자식 PK에 포함 |
| 형식적 분류 | 비식별자 관계 | 부모 식별자가 자식 일반속성(FK)으로 포함 |

---

### 4.2 슈퍼타입/서브타입 관계

| 구분 | 설명 |
| --- | --- |
| 슈퍼타입 | 공통 속성을 일반화한 상위 엔터티 |
| 서브타입 | 개별 속성으로 특수화한 하위 엔터티 |

| 서브타입 간 교집합 | 관계 유형 |
| --- | --- |
| 없음 | Exclusive 관계 |
| 있음 | Inclusive 관계 |

#### 슈퍼/서브타입 물리 변환 방식

| 방식 | 설명 | 적합한 경우 |
| --- | --- | --- |
| One to One Type | 개별 테이블로 분리 유지 | 개별 트랜잭션 중심 |
| Plus Type | 슈퍼+서브타입 결합 테이블 | 조인 비용이 큰 경우 |
| Single Type (All in One) | 전체를 단일 테이블로 통합 | 전체 조회가 빈번한 경우 |

## **5. 조인(Join) 예시**

### 5.1 원본 테이블

**[부서]**

| 부서번호 | 부서명 |
| --- | --- |
| 100 | 개발부 |
| 101 | 총무부 |
| 102 | 판매부 |

**[사원]**

| 사원번호 | 사원명 | 부서번호 |
| --- | --- | --- |
| 10001 | 이채원 | 100 |
| 10002 | 송민경 | 101 |
| 10003 | 김진호 | 102 |
| 10004 | 오세형 | 100 |

사원번호, 사원명, 부서명을 한 번에 조회하려면 `[사원]`과 `[부서]`를 `부서번호`로 조인한다.

### 5.2 조인 결과

| 사원번호 | 사원명 | 부서번호 | 부서명 |
| --- | --- | --- | --- |
| 10001 | 이채원 | 100 | 개발부 |
| 10002 | 송민경 | 101 | 총무부 |
| 10003 | 김진호 | 102 | 판매부 |
| 10004 | 오세형 | 100 | 개발부 |

### 5.3 조인 처리 흐름

| 단계 | 설명 |
| --- | --- |
| 1 | `[사원]`에서 대상 사원번호 행 조회 |
| 2 | 해당 행에서 사원명 확인 |
| 3 | 해당 행에서 부서번호 확인 |
| 4 | `[부서]`에서 같은 부서번호 행 조회 |
| 5 | `[사원]`의 사원정보 + `[부서]`의 부서명 결합 출력 |

> 위 과정에서 **3~4번이 조인 핵심**이며, 매핑키는 `부서번호`이다.  
> 관계가 없는 테이블 간 조인은 일반적으로 비즈니스 의미가 없다.

## **6. 자기 참조 관계**

| 항목 | 내용 |
| --- | --- |
| 정의 | 하나의 테이블에서 한 속성이 같은 테이블의 다른 행을 참조하는 관계 |
| 모델 형태 | 계층형 데이터 모델 |
| 조회 방식 | 계층형 질의, 셀프 조인(Self Join) |

```text
[사원]
사원ID (PK)
상위사원ID (FK -> 사원.사원ID)
사원명
```

## **7. 상호 배타적 관계(아크 관계, Arc)**

| 항목 | 내용 |
| --- | --- |
| 정의 | 두 개 이상의 부모 후보가 있으나 실제로는 그중 하나만 참조 가능한 관계 |
| 별칭 | 아크(Arc) 관계 |
| ERD 표현 | 보통 괄호 형태의 Arc 선으로 표시 |

```text
[결제]
결제ID
고객개인ID (FK, nullable)
고객법인ID (FK, nullable)

규칙: 고객개인ID, 고객법인ID 중 하나만 값이 존재해야 함
```
